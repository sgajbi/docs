# 01 - Commercial Model Fundamentals for Wealth Software

## 1. What a commercial model is

A commercial model defines how a product creates value, charges for that value, contracts with customers, delivers the service, supports production usage and grows over time.

For enterprise wealth software, the commercial model must answer:

- What business problem does the product solve?
- Who is the buyer?
- Who is the user?
- Who approves the budget?
- What is the value driver?
- What is charged?
- How is usage measured?
- How is risk controlled?
- How is support provided?
- How is the relationship renewed or expanded?

## 2. Wealth software buyer groups

| Buyer / stakeholder | What they care about |
|---|---|
| CIO / CTO | Architecture, integration, resilience, cost, vendor risk |
| COO / operations | Process efficiency, exception reduction, operational control |
| Head of advisory | Advisor productivity, client outcomes, suitability |
| Portfolio management | mandate controls, analytics, rebalancing, model portfolios |
| Reporting head | statement quality, automation, auditability, client trust |
| Risk / compliance | controls, audit trail, regulatory alignment |
| Procurement | commercial terms, vendor due diligence, negotiation |
| Legal | liability, data rights, IP, audit, termination |
| Security | data protection, IAM, encryption, incident process |
| Finance | ROI, TCO, budget impact, renewal discipline |
| End users | usability, speed, trust, workflow fit |

## 3. Common value themes in wealth platforms

| Value theme | Example |
|---|---|
| Revenue enablement | More advisors can serve more clients |
| Productivity | Reduce manual portfolio review preparation |
| Risk reduction | Prevent unsuitable proposals or mandate breaches |
| Cost reduction | Retire legacy reporting or vendor platform |
| Control improvement | Better lineage, audit, reconciliations |
| Client experience | More timely and explainable reporting |
| Speed to market | Faster country/product rollout |
| Platform modernization | API/event/cloud-native architecture |
| AI differentiation | Copilot, personalization, decision intelligence |
| Data quality | Source ownership, exception handling, traceability |

## 4. Commercial model components

| Component | Definition |
|---|---|
| Pricing metric | What the customer pays against |
| Packaging | How capabilities are grouped |
| Contract term | Length and renewal model |
| Deployment model | SaaS, private cloud, self-hosted, managed service |
| Support tier | Standard, premium, mission-critical |
| Implementation fee | One-time setup / migration / integration services |
| Usage limits | API calls, users, portfolios, AUM, reports |
| Expansion path | More users, modules, entities, regions |
| SLA | Operational commitments |
| Exit terms | Data export, termination, transition assistance |

## 5. Common commercial models

| Model | Description |
|---|---|
| Subscription SaaS | Recurring fee for access to software |
| Per-user pricing | Fee based on named or active users |
| Per-portfolio pricing | Fee based on number of portfolios/accounts |
| AUM-based pricing | Fee based on assets supported |
| Module-based pricing | Fee by product capability/module |
| Usage-based pricing | Fee by API call, report, calculation, workflow |
| Enterprise license | Broad usage rights for an agreed enterprise scope |
| Managed service | Vendor operates capability for client |
| Professional services | Implementation, migration, customization |
| Hybrid model | Combination of subscription + usage + services |

## 6. Bank-friendly commercial principles

Banks prefer commercial models that are:

- predictable
- auditable
- scalable
- easy to allocate internally
- aligned to value
- not punitive for successful adoption
- clear on included support
- transparent on optional services
- clear on data and usage rights
- resilient to organizational restructuring
- compatible with procurement and finance processes

## 7. Product versus service

A software company must avoid becoming only a consulting shop.

| Product business | Service business |
|---|---|
| Repeatable capability | Custom delivery |
| Scalable margins | People-heavy margins |
| Configurable | Bespoke |
| Versioned roadmap | Project-specific roadmap |
| Supportable at scale | Hard to support broadly |
| Reusable documentation | Client-specific knowledge |

Enterprise clients may request customization. The commercial model should protect the product core while allowing controlled configuration.

## 8. Wealth-specific commercial sensitivity

Wealth platforms often serve multiple countries, booking centres and legal entities.

Commercial design should consider:

- per-country rollout
- per-booking-centre pricing
- per-legal-entity contracting
- client segment differences
- advisory versus DPM usage
- internal chargeback
- regulatory data residency
- different product universes by market
- different support hours by region
- migration from legacy systems

## 9. Key takeaway

A strong commercial model converts technical capability into a bank-buyable business outcome.
