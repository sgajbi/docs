# 02 - SaaS Pricing, Packaging, Licensing and Monetization Models

## 1. Pricing design goals

Pricing should balance:

- customer value
- cost to serve
- buyer predictability
- expansion potential
- implementation complexity
- support intensity
- procurement simplicity
- margin sustainability
- competitive positioning

## 2. Pricing metric options for wealth software

| Metric | Good for | Watchouts |
|---|---|---|
| Named user | Advisor workbench, operations tools | Shelfware objections |
| Active user | Usage-sensitive teams | Requires accurate activity definition |
| Portfolio/account count | Portfolio analytics/reporting | Need rules for dormant/test accounts |
| AUM supported | DPM/reporting platforms | AUM volatility affects revenue |
| Report generated | Reporting engines | May discourage usage |
| API calls | Developer/data platforms | Hard for business users to predict |
| Calculations run | Analytics engines | Can create cost anxiety |
| Module | Enterprise products | Need clear module boundaries |
| Country/booking centre | Multi-country wealth rollouts | May slow expansion |
| Enterprise license | Large banks | Requires scope discipline |
| Outcome-based | Strong ROI linkage | Hard to measure and audit |

## 3. Packaging model

A good package structure:

```text
Core Platform
+ Product Data / Instrument Master
+ Portfolio Analytics
+ Reporting
+ Advisory Workbench
+ Mandate / DPM
+ AI Copilot
+ Operations / Reconciliation
+ Governance / Audit
```

Example:

| Package | Target buyer | Included capabilities |
|---|---|---|
| Foundation | Smaller platform team | Core APIs, data model, entitlements, audit |
| Analytics | Portfolio analytics teams | performance, risk, exposure, benchmark |
| Advisory | RM/advisor teams | portfolio review, proposal, suitability |
| DPM | Discretionary PM teams | mandates, drift, rebalance, restrictions |
| Reporting | Client reporting teams | statements, PDFs, archives, exception panels |
| AI | Innovation/advisory/productivity teams | copilot, RAG, explanation, workflow assist |
| Enterprise | Large banks | All modules, advanced support, multi-country |

## 4. Good-better-best packaging

| Tier | Description |
|---|---|
| Essential | Core product, limited configuration, standard support |
| Professional | Broader features, APIs, integrations, workflow support |
| Enterprise | Multi-entity, audit, SLAs, advanced security, premium support |
| Regulated Enterprise | Highest controls, dedicated environments, DR, audit evidence |

## 5. Module pricing

Module pricing works well when capabilities have separate buyers.

Example wealth-platform suite modules:

| Module | Possible pricing driver |
|---|---|
| wealth-platform suite Core | instruments, accounts, prices, FX data volume |
| wealth-platform suite Performance | portfolios/calculation volume |
| wealth-platform suite Risk | portfolios/exposure/risk models |
| wealth-platform suite Report | reports generated / portfolios / entities |
| wealth-platform suite Advise | advisors / proposals / users |
| wealth-platform suite Manage | DPM portfolios / mandates |
| wealth-platform suite AI | users + usage + governance tier |
| wealth-platform suite Archive | storage + retrieval + retention |
| wealth-platform suite Gateway | API traffic + connected systems |

## 6. Usage-based pricing

Usage-based pricing is attractive when platform usage varies materially.

Possible meters:

- API calls
- report renders
- portfolio calculations
- AI tokens / AI tasks
- data storage
- archive retrieval
- workflow executions
- benchmark calculations
- risk scenario runs
- integration connectors

Controls needed:

- usage dashboard
- quota alerts
- overage terms
- fair-use policy
- internal chargeback export
- test/non-prod exclusion rules
- batch pricing rules
- AI cost controls

## 7. Enterprise license

Enterprise licenses simplify procurement but require tight scoping.

Define:

- legal entities covered
- countries / booking centres
- user groups
- portfolios/accounts included
- modules included
- environments included
- support tier
- API/calc/report limits
- AI usage limits
- affiliate usage rights
- renewal uplift
- expansion triggers

## 8. Implementation fees

Banks expect implementation cost.

Implementation fees may cover:

- discovery
- architecture
- integration design
- configuration
- data mapping
- migration
- UAT support
- performance testing
- security support
- training
- cutover support
- hypercare

Avoid unlimited implementation scope under a low fixed fee.

## 9. Discounts

Discounts should be controlled:

| Discount type | Use carefully |
|---|---|
| First-year discount | Land-and-expand |
| Multi-year discount | Commitment |
| Volume discount | Large portfolio/user scale |
| Bundle discount | Multiple modules |
| Strategic logo discount | Reference customer |
| Pilot credit | Convert pilot to production |

Always define renewal baseline so discounts do not permanently destroy pricing.

## 10. Key takeaway

The best pricing metric is one the customer understands, the vendor can measure, and both sides see as fair.
