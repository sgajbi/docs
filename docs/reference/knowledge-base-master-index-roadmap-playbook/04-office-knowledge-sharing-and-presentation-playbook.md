# 04 - Office Knowledge Sharing and Presentation Playbook

## 1. Purpose

This file helps convert the knowledge base into office learning sessions.

The goal is to explain topics clearly to:

- business analysts
- developers
- QA
- operations
- product owners
- architects
- reporting teams
- advisory stakeholders
- management

## 2. Session types

| Session type | Duration | Purpose |
|---|---:|---|
| Product primer | 30 min | Explain one product family |
| Platform modelling deep dive | 45-60 min | Transactions, positions, valuation, lifecycle |
| Analytics session | 60 min | Performance, attribution, risk, reporting |
| Architecture session | 60 min | APIs, events, domains, data flows |
| Control session | 45 min | governance, QA, audit, reconciliation |
| practitioner-style teach-back | 20 min | Practice concise explanation |
| Case study workshop | 90 min | Work through realistic scenario |

## 3. Standard presentation structure

Use this structure for any product:

```text
1. What the product is
2. Why clients use it
3. How payoff / income works
4. Main risks
5. Lifecycle events
6. Transaction and position model
7. Valuation and pricing
8. Performance / risk / reporting treatment
9. Advisory and suitability considerations
10. Platform implementation and QA scenarios
```

## 4. Example: structured notes session

Agenda:

1. What is a note?
2. Plain note vs structured note
3. Components: debt + derivative + issuer risk
4. FCN, ELN, autocall, CLN, DCI
5. Lifecycle: subscription, coupon, observation, barrier, maturity, physical delivery
6. Position model: note as legal holding, underlying as look-through exposure
7. Valuation: issuer quote, model price, barrier/volatility/correlation
8. Reporting: income, risk, issuer exposure, payoff scenarios
9. Advisory: high coupon is risk compensation
10. QA: barrier breach, autocall, physical delivery, missing price

## 7. Teach-back method

After studying a topic, practice:

- 2-minute explanation for senior leader
- 5-minute explanation for BA/developer
- 10-minute platform design explanation
- 1 worked numerical example
- 1 reporting example
- 1 QA scenario
- 1 practitioner answer

## 8. Knowledge-sharing quality checklist

Before presenting:

- does the session have a clear audience?
- does it explain business purpose?
- does it include examples?
- does it cover platform modelling?
- does it cover advisory/reporting impact?
- does it include edge cases?
- does it include diagrams or tables?
- does it end with key takeaways?
- does it avoid duplicated explanations?
- does it link to source packs?
