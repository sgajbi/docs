# 02 - Outsourcing, Procurement and Bank-Buyable Software Lifecycle

## 1. Procurement lifecycle

A typical bank procurement lifecycle includes:

| Stage | Key questions |
|---|---|
| Business need | What problem are we solving? |
| Sourcing | Which vendors can solve it? |
| RFI / RFP | Can they meet functional, technical, security and commercial needs? |
| Proof of concept | Does the product work in realistic conditions? |
| Vendor due diligence | Is the vendor safe, stable and controlled? |
| Architecture review | Can it integrate with bank systems safely? |
| Security review | Does it meet security and privacy standards? |
| Legal review | Are contract terms acceptable? |
| Risk committee approval | Is the residual risk acceptable? |
| Implementation | Can it be delivered safely? |
| Production readiness | Is support, monitoring, resilience and recovery ready? |
| Go-live | Controlled deployment and sign-off |
| Ongoing monitoring | Periodic review, incidents, SLA, controls |
| Exit / renewal | Decide whether to continue, remediate or exit |

## 2. Business case

A vendor proposal should explain:

- business capability enabled
- target users
- expected benefits
- cost and licensing model
- implementation effort
- integration scope
- data dependencies
- operating model impact
- risk and control implications
- expected time to value
- exit and fallback options

## 3. RFI / RFP evaluation dimensions

| Dimension | What to evaluate |
|---|---|
| Functional fit | Features, workflows, reporting, localization |
| Data fit | Data model, APIs, export, lineage, quality controls |
| Security | IAM, encryption, logging, vulnerability management |
| Privacy | PII processing, data residency, deletion, consent |
| Resilience | Availability, DR, RTO/RPO, incident management |
| Integration | APIs, events, SSO, data feeds, webhooks |
| Operations | Support model, SLAs, escalation, monitoring |
| Compliance | Audit reports, certifications, regulatory readiness |
| Commercial | Pricing, licensing, usage limits, renewal terms |
| Vendor stability | Financial health, ownership, roadmap, references |
| AI governance | Explainability, audit, evaluation, guardrails, human oversight |
| Exit | Data portability, termination support, escrow if applicable |

## 4. Bank-buyable software evidence pack

A credible vendor should be able to provide:

- company profile and ownership structure
- financial statements or proof of viability
- architecture overview
- security whitepaper
- data-flow diagram
- data processing agreement
- cloud hosting details
- subcontractor list
- business continuity and DR plan
- incident response process
- vulnerability management process
- penetration test summary
- SOC 2 / ISO 27001 / equivalent evidence where available
- SDLC and release management process
- support model and SLA catalogue
- audit and compliance evidence
- insurance coverage details
- privacy policy and data retention schedule
- AI governance documentation if AI is used
- exit plan and data deletion process

## 5. Material outsourcing considerations

A relationship is more likely to be material or critical if:

- it supports client-facing services
- it affects regulated activities
- failure would materially disrupt operations
- it handles confidential or sensitive data
- it is hard to substitute
- it supports risk, finance, reporting or compliance
- it supports AI-based advisory or decision workflows
- it has deep integration with core platforms
- it is used across multiple business units or countries

## 6. Approval forums

Depending on size and criticality, approvals may involve:

- business sponsor
- architecture review board
- technology risk
- information security
- privacy / data protection
- compliance
- legal
- procurement
- operational risk
- outsourcing committee
- cloud risk committee
- AI / model-risk committee
- country / booking-centre governance
- finance / commercial approval

## 7. Production onboarding gates

Before go-live, require:

- signed contract
- approved architecture
- completed security assessment
- data classification approval
- user access model
- audit logging enabled
- monitoring and alerting
- support and escalation model
- DR evidence
- runbook
- operational acceptance
- reconciliation checks
- data migration sign-off
- user acceptance testing
- penetration/vulnerability closure plan
- service owner and control owner assignment

## 8. Ongoing monitoring

Monitoring should cover:

- SLA performance
- incidents and outages
- change notifications
- vulnerability remediation
- financial health
- control attestations
- audit reports
- subcontractor changes
- data-location changes
- business continuity tests
- regulatory changes
- complaints and client impact
- support quality
- AI evaluation drift if applicable

## 9. Renewal decision

At renewal, ask:

- Is the vendor still strategically aligned?
- Are controls still adequate?
- Are SLAs being met?
- Is pricing still fair?
- Are there unresolved incidents or audit issues?
- Has the product roadmap changed?
- Has the regulatory environment changed?
- Are alternatives available?
- Is exit more appropriate than renewal?

## 10. Key takeaway

Procurement is not just vendor selection. It is risk acceptance, operational design and accountability creation.
