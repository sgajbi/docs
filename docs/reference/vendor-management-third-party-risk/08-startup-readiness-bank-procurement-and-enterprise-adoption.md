# 08 - Startup Readiness, wealth-platform suite, Bank Procurement and Enterprise Adoption

## 1. Why this matters for bank-buyable software

For a wealth-platform product to become bank-buyable, it must be more than good code.

It must show:

- professional company setup
- strong product documentation
- secure engineering practices
- clear architecture
- reliable operations
- data protection
- auditability
- implementation maturity
- support model
- integration readiness
- exit readiness
- credible roadmap
- evidence of controls

## 2. Bank-buyable maturity levels

| Level | Description |
|---|---|
| Level 0 | Prototype only |
| Level 1 | Working product with documentation |
| Level 2 | Secure product with CI/CD, tests and support model |
| Level 3 | Enterprise-ready product with audit evidence and controls |
| Level 4 | Bank-buyable platform with certifications, SLAs, DR, security evidence |
| Level 5 | Regulated-sector proven platform with multiple enterprise deployments |

## 3. Enterprise evidence pack

Each wealth-platform application should maintain:

- README
- architecture document
- API specification
- data model
- threat model
- control matrix
- test strategy
- release process
- support runbook
- incident process
- deployment guide
- data retention and deletion guide
- observability guide
- security controls
- dependency/SBOM evidence
- privacy impact notes
- product capability matrix
- known limitations
- roadmap
- source notes and assumptions

## 4. Startup corporate readiness

A startup selling to banks should prepare:

- incorporated legal entity
- clean ownership structure
- founder/management details
- audited or management accounts when available
- insurance coverage
- standard MSA / SaaS agreement
- DPA template
- security policy
- privacy policy
- business continuity plan
- incident response plan
- employment / contractor confidentiality clauses
- IP ownership assignment
- open-source license review
- customer support model

## 5. Product readiness

For each product:

- clear problem statement
- target customer
- target users
- supported use cases
- unsupported use cases
- integration patterns
- deployment model
- data flow
- security model
- API contract
- user permissions
- reporting outputs
- audit logs
- error handling
- scalability assumptions
- performance benchmarks
- QA evidence

## 6. Security readiness

At minimum:

- SSO / OAuth / OIDC support
- MFA for admin users
- RBAC / ABAC
- encryption in transit and at rest
- secrets management
- vulnerability scanning
- dependency scanning
- container image scanning
- secure CI/CD
- audit logs
- tenant isolation design
- backup and restore process
- incident response process
- secure coding standard
- security test evidence

## 7. AI readiness for AI-enabled wealth applications

For AI apps:

- approved knowledge sources
- RAG grounding
- citation requirements
- hallucination controls
- prompt injection controls
- PII redaction
- tool-use approval
- human-in-the-loop workflows
- evaluation suite
- conversation audit trail
- model version tracking
- no unapproved training on client data
- action limits and permissions

## 8. Enterprise sales readiness

A bank buyer will ask:

- Why this product?
- Why your company?
- Why now?
- What is differentiated?
- Who else uses it?
- Can it scale?
- Can it pass security?
- Can it integrate?
- Can it operate under SLA?
- Can we audit it?
- Can we exit safely?
- Can you support us for years?

## 9. Practical wealth-platform roadmap

| Phase | Focus |
|---|---|
| Phase 1 | Documentation, architecture, tests, CI/CD |
| Phase 2 | Security controls, observability, audit logs |
| Phase 3 | Deployment patterns, runbooks, support model |
| Phase 4 | Enterprise evidence pack and demo environments |
| Phase 5 | Pilot with controlled non-production client data |
| Phase 6 | Certification and procurement readiness |
| Phase 7 | Enterprise onboarding and SLA operations |

## 10. Key takeaway

To sell to banks, wealth-platform suite must be designed as a controlled enterprise product from the beginning.
