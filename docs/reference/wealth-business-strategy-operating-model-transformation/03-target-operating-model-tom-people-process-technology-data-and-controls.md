# 03 - Target Operating Model: People, Process, Technology, Data and Controls

## 1. What a TOM is

A Target Operating Model defines how the future organization will operate.

It describes the future state across:

- clients and channels
- products and services
- people and roles
- processes and workflows
- technology platforms
- data ownership
- controls and governance
- vendor and sourcing model
- metrics and management rhythm

## 2. TOM dimensions

| Dimension | Key questions |
|---|---|
| Client | Which segments and journeys are supported? |
| Product | What products and services are offered? |
| Channel | Advisor, digital, client portal, API, operations |
| People | Roles, skills, ownership, responsibilities |
| Process | Standardized workflows, exceptions, approvals |
| Technology | Platforms, APIs, events, data stores |
| Data | Sources, lineage, quality, ownership |
| Controls | Suitability, audit, reconciliation, security |
| Organization | Team structure, capability ownership |
| Vendor | Build/buy/partner decisions |
| Metrics | KPIs, OKRs, risk indicators |

## 3. Current state versus target state

| Area | Current-state symptom | Target-state pattern |
|---|---|---|
| Product data | inconsistent taxonomy | governed product master |
| Portfolio reporting | manual adjustments | reproducible report snapshots |
| Performance | country-specific logic | common engine with localization |
| Advisory | advisor judgement not captured | workflow with evidence and suitability |
| DPM | manual drift monitoring | automated mandate and rebalance controls |
| Data quality | late discovery | embedded DQ and exception dashboards |
| AI | ad hoc experimentation | governed AI service with evaluation |
| Architecture | point-to-point | API/event domain platform |
| Operations | spreadsheet controls | workflow, reconciliation, audit trail |
| Migration | manual sign-off | automated regression and parallel run evidence |

## 4. People and role model

| Role | Responsibility |
|---|---|
| Business capability owner | Owns outcome and process |
| Product owner | Owns backlog and feature value |
| Platform architect | Owns architecture coherence |
| Domain architect | Owns domain model and integration |
| Data owner | Owns data quality and definition |
| Control owner | Owns control design and evidence |
| Engineering lead | Owns build quality and delivery |
| Operations lead | Owns support and exception handling |
| AI product owner | Owns AI use cases and governance |
| Model risk/compliance | Owns AI/model risk approval |
| Release manager | Owns controlled deployment |
| Country lead | Owns localization and country readiness |

## 5. Process design

Target processes should be:

- standardized where possible
- configurable where required
- auditable
- exception-aware
- source-backed
- measurable
- integrated with approvals
- resilient to missing data
- clear on ownership
- capable of producing evidence

## 6. Technology model

Target architecture should prefer:

- domain-aligned services
- clear system of record
- event-driven ingestion where useful
- APIs for consumption
- snapshot-based reporting
- calculation engines with versioning
- immutable audit logs
- observability
- automated reconciliation
- secure identity and entitlements
- cloud-native deployment where appropriate
- AI services with controlled tool use

## 7. Data model

Target data principles:

- data domains have owners
- reference data has golden source
- market data has pricing hierarchy
- transaction data is immutable or correction-based
- positions are derived and reconcilable
- reports are reproducible
- AI outputs are grounded and traceable
- data quality is measured
- lineage is visible
- country differences are explicit

## 8. Control model

Controls should be embedded in workflows:

| Control type | Example |
|---|---|
| Preventive | block unsuitable product |
| Detective | identify stale price |
| Corrective | raise remediation workflow |
| Compensating | manual review when automation unavailable |
| Evidential | store approval and calculation version |
| Supervisory | dashboard and review attestation |

## 9. TOM deliverables

A TOM pack should include:

- current-state assessment
- target-state narrative
- capability map
- process maps
- role/responsibility matrix
- system landscape
- data domain model
- control framework
- vendor model
- migration approach
- roadmap
- cost/benefit case
- KPI dashboard
- transition plan

## 10. Key takeaway

A TOM makes transformation operationally real. It says not only what will change, but how the future business will run.
