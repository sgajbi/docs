# 05 - Advisory, DPM, Reporting, Analytics and AI Operating Model

## 1. Why this layer matters

The modern wealth operating model is no longer only transaction processing. It is decision support.

Core decision areas:

- What should advisor discuss with client?
- Is portfolio aligned to client objectives?
- Is product suitable?
- Is DPM mandate within limits?
- What caused performance?
- What risks are concentrated?
- What should be rebalanced?
- What should be explained in client report?
- What exception needs attention?
- How can AI assist safely?

## 2. Advisory operating model

| Capability | Operating model need |
|---|---|
| Client profiling | updated suitability and goals |
| Product recommendation | APU, target market, risk rating |
| Portfolio review | analytics, risk, income, performance |
| Proposal generation | target allocation and trade simulation |
| Suitability checks | documented rationale and controls |
| Client consent | evidence and communication record |
| Follow-up | tasks, notes, reminders |

## 3. DPM operating model

| Capability | Operating model need |
|---|---|
| Model portfolios | CIO office / investment committee ownership |
| Mandates | rules, tolerances, restrictions |
| Drift monitoring | automated checks and escalation |
| Rebalancing | trade generation and approval |
| Performance | benchmark-relative reporting |
| Breach management | evidence, remediation, client communication |
| Governance | committee, investment view, model changes |

## 4. Reporting operating model

| Capability | Operating model need |
|---|---|
| Report data | snapshot and lineage |
| Report templates | product/country/client-segment variants |
| Exception handling | stale/missing data disclosures |
| Review and approval | maker-checker and sign-off |
| Delivery | digital, email, archive, RM delivery |
| Restatement | correction process and audit |
| Client explanation | advisor-ready narrative |

## 5. Analytics operating model

Analytics requires:

- calculation methodology ownership
- versioned formulas
- source data quality
- benchmark governance
- product-specific treatment
- exception policy
- recomputation policy
- control evidence
- report integration
- advisor explanation

## 6. AI operating model

AI should support humans, not bypass controls.

AI use cases:

| Use case | Human oversight |
|---|---|
| Portfolio summary | advisor reviews before client use |
| Product explanation | must use approved sources |
| Performance narrative | must cite calculation and data |
| Suitability assist | must not override rules |
| Rebalance explanation | PM/advisor approval |
| Operations triage | operations owner confirms action |
| Client message draft | RM approval before sending |

## 7. AI guardrails

- approved knowledge sources only
- retrieval grounding
- citations
- no unsupported recommendation
- no autonomous order placement
- no client communication without approval
- entitlements enforced
- PII protection
- prompt injection protection
- evaluation suite
- audit logging
- model version tracking

## 8. Integrated operating model

The ideal future flow:

```text
Data quality confirms portfolio data
-> analytics calculates performance/risk
-> mandate engine identifies drift/breach
-> AI generates grounded explanation
-> advisor reviews and edits
-> suitability engine checks recommendation
-> client proposal/report generated
-> approval and consent captured
-> order workflow initiated
-> audit evidence stored
```

## 9. Key takeaway

The future wealth operating model is human-led, data-driven, AI-assisted and control-by-design.
