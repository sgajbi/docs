# AI Documentation, User Guidance, Training, and Change Management

## Purpose

This file explains how to document and roll out AI capabilities responsibly.

Users need to understand what AI can do, what it cannot do, and how to use it safely.

---

## User Guidance

Guidance should explain:

- purpose
- supported use cases
- unsupported use cases
- data used
- output limitations
- human review expectations
- source/citation meaning
- feedback mechanism
- escalation path
- privacy notes

---

## In-Product Guidance

Use:

- generated output labels
- review-required banners
- source links
- confidence/limitation notes
- stale-source warnings
- permission-blocked messages
- unsafe-use warnings
- feedback buttons

---

## Training

Training should cover:

- AI is assistive
- source grounding
- human accountability
- hallucination risk
- sensitive-data rules
- prompt safety
- prohibited uses
- feedback process
- escalation path

---

## Change Management

Rollout should include:

- pilot group
- feedback loop
- risk review
- adoption metrics
- support plan
- training materials
- known limitations
- phased expansion
- rollback plan

---

## Documentation Anti-Patterns

| Anti-Pattern | Risk |
|---|---|
| AI feature launched without guidance | Misuse. |
| Users not told limitations | Over-trust. |
| No prohibited-use list | Risky behavior. |
| No feedback path | Quality issues persist. |
| Training ignores sensitive data | Data leakage. |
| Pilot feedback not reviewed | Poor adoption. |

---

## Review Checklist

- Is user guidance written?
- Are limitations clear?
- Are prohibited uses listed?
- Is human review explained?
- Are citations explained?
- Is feedback mechanism available?
- Is training planned?
- Is rollout phased?
- Is support model ready?
- Is rollback plan defined?

---

## Summary

AI adoption is a change-management problem as much as a technical problem.

Users need guidance, transparency, and support.
