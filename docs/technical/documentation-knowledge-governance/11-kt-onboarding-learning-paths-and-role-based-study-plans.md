# KT, Onboarding, Learning Paths, and Role-Based Study Plans

## Purpose

This file explains how to create knowledge-transfer and onboarding systems for enterprise engineering teams.

KT should help people understand the platform, not just listen to walkthroughs.

---

## Onboarding Goals

A new joiner should learn:

- business context
- domain concepts
- platform map
- service ownership
- architecture boundaries
- local setup
- validation commands
- test strategy
- CI/CD workflow
- security rules
- observability and runbooks
- contribution process
- support model

---

## Role-Based Learning Paths

| Role | Learning Focus |
|---|---|
| Developer | repo setup, architecture, tests, APIs, Git workflow. |
| Senior engineer | boundaries, CI/CD, resilience, observability, review. |
| Architect | context maps, ADRs, service ownership, data flows. |
| QA engineer | test strategy, certification, evidence, regression. |
| SRE | runtime, dashboards, alerts, runbooks, incidents. |
| Product technologist | capabilities, status, domain, supported claims. |

---

## KT Session Structure

A good KT session includes:

1. context
2. capability ownership
3. architecture map
4. main flows
5. API/contracts
6. data ownership
7. runtime
8. testing
9. CI/CD
10. observability
11. security
12. known gaps
13. hands-on exercise

---

## Hands-On Learning

Good exercises:

- run service locally
- execute test suite
- trace one API flow
- update one doc safely
- add one unit test
- inspect one runbook
- review one PR
- read one ADR
- follow one incident scenario

---

## Onboarding Anti-Patterns

| Anti-Pattern | Impact |
|---|---|
| KT is only meetings | Knowledge not retained. |
| No reading path | New joiners overwhelmed. |
| No hands-on tasks | Shallow understanding. |
| Outdated setup docs | Frustration. |
| No domain glossary | Business confusion. |
| No architecture map | Code navigation slow. |
| No review checklist | Quality inconsistent. |

---

## Review Checklist

- Is onboarding path documented?
- Is it role-based?
- Are hands-on tasks included?
- Are docs current?
- Are diagrams included?
- Are common questions answered?
- Is ownership clear?
- Is progress measurable?
- Is KT recorded as durable docs?

---

## Summary

Good KT creates independent contributors.

The goal is not to transfer facts once. The goal is to build a repeatable learning system.
