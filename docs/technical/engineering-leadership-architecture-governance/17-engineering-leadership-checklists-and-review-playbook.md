# Engineering Leadership Checklists and Review Playbook

## Purpose

This file provides practical checklists for engineering leads, architects, and senior engineers.

Use it during architecture reviews, PR reviews, release reviews, production-readiness checks, roadmap planning, and team growth discussions.

---

## Architecture Review Checklist

- [ ] Problem statement clear.
- [ ] Goals and non-goals clear.
- [ ] Service ownership clear.
- [ ] Domain boundaries respected.
- [ ] API/event/data contracts defined.
- [ ] Data ownership and lineage clear.
- [ ] Security model defined.
- [ ] Failure modes covered.
- [ ] Testing strategy defined.
- [ ] Rollout and rollback defined.
- [ ] ADR/RFC created if needed.

---

## Service Ownership Checklist

- [ ] Owner team identified.
- [ ] Service mission clear.
- [ ] Owned capabilities listed.
- [ ] Non-owned capabilities listed.
- [ ] Public contracts listed.
- [ ] Dependencies listed.
- [ ] Runtime owner clear.
- [ ] Support owner clear.
- [ ] Documentation owner clear.
- [ ] Quality posture visible.

---

## PR Leadership Review Checklist

- [ ] Scope reviewable.
- [ ] Architecture impact understood.
- [ ] Tests meaningful.
- [ ] CI evidence sufficient.
- [ ] Security impact covered.
- [ ] Observability impact covered.
- [ ] Docs updated.
- [ ] Migration/runtime impact covered.
- [ ] Rollback or mitigation clear.
- [ ] Follow-up risks documented.

---

## Production Readiness Checklist

- [ ] Runtime evidence available.
- [ ] Health/readiness meaningful.
- [ ] Dashboards ready.
- [ ] Alerts actionable.
- [ ] Runbooks current.
- [ ] Security findings resolved or accepted.
- [ ] Migration tested.
- [ ] Rollback plan defined.
- [ ] Support owner ready.
- [ ] Known risks communicated.

---

## Technical Debt Checklist

- [ ] Debt type classified.
- [ ] Impact described.
- [ ] Evidence linked.
- [ ] Owner assigned.
- [ ] Priority agreed.
- [ ] Smallest improvement slice defined.
- [ ] Tests protect refactor.
- [ ] Timeline or trigger defined.
- [ ] Status reviewed regularly.

---

## Quality Scorecard Checklist

- [ ] Architecture assessed.
- [ ] API contracts assessed.
- [ ] Testing assessed.
- [ ] CI/CD assessed.
- [ ] Security assessed.
- [ ] Observability assessed.
- [ ] Runtime assessed.
- [ ] Documentation assessed.
- [ ] Operations assessed.
- [ ] Improvement actions defined.

---

## Stakeholder Communication Checklist

- [ ] Audience clear.
- [ ] Outcome clear.
- [ ] Status evidence-backed.
- [ ] Risks explained in business terms.
- [ ] Dependencies visible.
- [ ] Decision needed clearly stated.
- [ ] Next milestone clear.
- [ ] Unsupported claims avoided.
- [ ] Follow-up owner clear.

---

## Team Growth Checklist

- [ ] KT path exists.
- [ ] Onboarding tasks defined.
- [ ] Engineers own meaningful slices.
- [ ] Seniors coach others.
- [ ] Review culture healthy.
- [ ] Standards documented.
- [ ] Production thinking taught.
- [ ] Knowledge risks reduced.
- [ ] Learning sessions active.

---

## Incident Learning Checklist

- [ ] Incident reviewed.
- [ ] Impact understood.
- [ ] Detection assessed.
- [ ] Runbook assessed.
- [ ] Root/contributing causes identified.
- [ ] Tests/gates improved.
- [ ] Alerts/dashboards improved.
- [ ] Architecture or code fixed.
- [ ] Actions owned and tracked.

---

## Engineering Lead Daily Questions

- What risk is currently hidden?
- What decision is blocked?
- What quality issue keeps repeating?
- What production signal needs attention?
- What team needs help?
- What should be simplified?
- What should be documented?
- What evidence is missing?
- What can be improved this week?

---

## Summary

Engineering leadership is repeatable judgment.

Use these checklists to make good judgment visible, teachable, and scalable.
