# Enterprise Buyer Expectations and Bank Procurement Readiness

## Purpose

This file explains what regulated enterprise buyers typically assess before procuring software.

Large banks and financial institutions evaluate not only features but supplier risk, security posture, operational readiness, integration cost, data protection, and long-term maintainability.

---

## Buyer Evaluation Areas

| Area | Buyer Question |
|---|---|
| Business value | What problem does the product solve and for whom? |
| Capability | What is supported today versus planned? |
| Architecture | How does it fit our estate? |
| Integration | What APIs, files, events, SSO, and data flows are needed? |
| Security | How are identity, access, secrets, and vulnerabilities handled? |
| Privacy | What client/personal data is processed and where? |
| Resilience | How does the service fail, recover, and scale? |
| Operations | Who supports it and what SLA applies? |
| Compliance | What evidence proves controls? |
| Exit | How do we retrieve data and terminate safely? |
| Commercial | What is licensed, priced, and contractually committed? |

---

## Procurement Evidence

Common evidence requested:

- company/supplier profile
- architecture overview
- security whitepaper
- data flow diagram
- data classification
- privacy posture
- SOC/ISO or equivalent evidence where available
- penetration/security assessment where available
- vulnerability management process
- incident response process
- BCP/DR posture
- SLA/support model
- third-party dependency list
- hosting/deployment model
- audit logs and access controls
- product roadmap
- contract and licensing model

---

## Procurement Readiness Levels

| Level | Description |
|---|---|
| L0 | Feature prototype only. |
| L1 | Product scope and architecture documented. |
| L2 | Security, deployment, support, and docs baseline ready. |
| L3 | Evidence-backed quality, controls, and operations. |
| L4 | Pilot-ready with synthetic proof and due-diligence pack. |
| L5 | Procurement-ready with mature evidence, support, contracts, and implementation model. |

---

## Buyer Concern Anti-Patterns

| Anti-Pattern | Impact |
|---|---|
| Cannot explain data flow | Security/privacy blocker. |
| No supported-feature matrix | Scope confusion. |
| No vulnerability management process | Security blocker. |
| No support SLA | Operations blocker. |
| No exit strategy | Procurement blocker. |
| No architecture diagrams | Integration uncertainty. |
| No evidence pack | Claims not trusted. |

---

## Review Checklist

- Is buyer value clear?
- Is supported scope clear?
- Is architecture documented?
- Is integration model documented?
- Is security evidence prepared?
- Is data processing documented?
- Is support model defined?
- Is SLA/SLO posture defined?
- Is exit plan defined?
- Is due-diligence pack ready?

---

## Summary

Enterprise procurement is risk evaluation.

A product is easier to buy when it reduces uncertainty for business, technology, security, operations, legal, and procurement stakeholders.
