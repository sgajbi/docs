# Commercial Packaging, Licensing, Pricing, and Contract Readiness

## Purpose

This file explains commercial-readiness topics that connect product engineering to enterprise buying.

Technical products need clear commercial packaging, licensing, and contract support.

---

## Product Packaging

Packaging may be by:

- module
- capability
- user role
- environment
- transaction volume
- asset/portfolio volume
- API usage
- report generation volume
- deployment model
- support tier

---

## Licensing Models

Examples:

| Model | Description |
|---|---|
| Subscription | Recurring fee. |
| Per user | Priced by named/active users. |
| Usage-based | Priced by API calls, reports, workflows, or compute. |
| Module-based | Priced by purchased capabilities. |
| Enterprise license | Broad usage within agreed scope. |
| Pilot license | Limited evaluation scope. |

---

## Contract Topics

Commercial contracts may need:

- scope
- license rights
- support obligations
- SLA
- data handling
- confidentiality
- security obligations
- audit rights
- subprocessor disclosure
- liability
- IP ownership
- termination
- exit assistance
- change management
- roadmap disclaimers

---

## Commercial Anti-Patterns

| Anti-Pattern | Impact |
|---|---|
| Pricing unrelated to buyer value | Sales friction. |
| License scope vague | Contract disputes. |
| Support tier unclear | Operational conflict. |
| Pilot scope undefined | Expectation mismatch. |
| Usage metrics not implemented | Billing dispute. |
| Roadmap promises in contract carelessly | Delivery/legal risk. |

---

## Review Checklist

- Is product packaging clear?
- Is license unit measurable?
- Is support tier defined?
- Is pilot scope clear?
- Are usage metrics available?
- Are contract obligations understood?
- Is exit assistance defined?
- Are roadmap commitments controlled?
- Are legal/commercial reviews planned?

---

## Summary

Commercial readiness should match technical reality.

Do not sell or contract what the product cannot evidence and support.
