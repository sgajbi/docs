# Incident Communication, Business Updates, and Status Pages

## Purpose

This file explains how to communicate during incidents.

Good incident communication reduces uncertainty, builds trust, and helps stakeholders make decisions.

---

## Communication Principles

Incident communication should be:

- timely
- factual
- concise
- audience-appropriate
- impact-focused
- non-blaming
- updated regularly
- clear about next update
- clear about workaround
- clear about recovery status

---

## Audience Types

| Audience | Needs |
|---|---|
| Users | What is impacted, workaround, expected update. |
| Business owners | Business impact, risk, timeline, decision needs. |
| Support | Known issue text, triage guidance, ticket handling. |
| Leadership | severity, impact, mitigation, risk, ETA if known. |
| Engineering | technical facts, dashboards, actions, owners. |
| Risk/security | data, privacy, regulatory, control impact. |

---

## Update Template

```text
Incident:
Severity:
Impact:
Started:
Current status:
Mitigation:
Workaround:
Risk/data impact:
Next update:
Owner:
```

Avoid false certainty.

---

## Status Page

A status page can show:

- current status
- affected services
- incident timeline
- updates
- workaround
- resolved state
- post-incident summary where appropriate

---

## Communication Anti-Patterns

| Anti-Pattern | Impact |
|---|---|
| Silence during incident | Stakeholder anxiety. |
| Overly technical business update | Message lost. |
| Promise ETA without confidence | Trust damage. |
| Blame language | Collaboration suffers. |
| Updates scattered across many chats | Confusion. |
| Known issue not shared with support | Ticket duplication. |
| No final resolution note | Unclear closure. |

---

## Review Checklist

- Are communication channels defined?
- Is communications lead assigned?
- Is update template available?
- Are audiences identified?
- Is business impact explained?
- Is workaround communicated?
- Are updates regular?
- Is sensitive detail controlled?
- Is closure communicated?
- Is post-incident summary planned?

---

## Summary

Incident communication is part of incident management.

A technically strong response can still fail if stakeholders are left guessing.
